Before we dive into use cases, let's define what we're talking about. An AI agent is not autocomplete, not a chatbot. The ramp-up reduction is dramatic. This is the best way to get comfortable with agents before using them to write code. ~1min

Your Agent Is Only as Good as Its Instructions

AGENTS.md · CLAUDE.md · The concept is universal

  • What to include: build/test/lint commands, code style rules, architectural decisions, dev environment quirks
  • Keep it SHORT (~100 lines). Too much guidance becomes non-guidance.
  • Think of it as onboarding a new hire: tell them the non-obvious stuff

The hierarchy:

Project root → Subdirectories → Personal overrides → Global → Enterprise
Now that you've seen what agents can do, here's how to make them better. A file the agent reads at session start. AGENTS.md in Droid, CLAUDE.md in Claude Code -- same idea. Keep it short. OpenAI tried a massive one and it failed. Think new-hire onboarding doc: only the non-obvious stuff. These nest -- subdirectory files load on demand, shared via git. ~1min

Skills Turn One-Time Solutions into Reusable Knowledge

Composable procedural knowledge for agents

  • Folders with instructions and scripts that teach agents specific jobs
  • Progressive disclosure -- only metadata loaded initially, full content on demand
  • Hundreds of skills without bloating context
  • Skills can include scripts as tools -- self-documenting, modifiable, live in the file system

Examples:

/fix-issue 1234 /incident-runbook /code-review
Skills are runbooks the agent follows. Not prompt templates -- structured procedures with scripts. The key innovation is progressive disclosure: the agent sees metadata for hundreds of skills but only loads the full content when it decides to use one. This keeps context clean. And it's not just engineers -- teams in finance, legal, and recruiting are building skills too. Skills complement MCP tools: MCP provides the connection to external services, skills provide the expertise for how to use them. ~1min

Use Case 2: Code Review & Testing

Low risk · High leverage

  • Agent as FIRST pass -- logic errors, security, style violations
  • Human review as SECOND pass -- business logic, architecture
  • Reduces cycle time by ~3.5 hours per PR
  • Can be configured as automated PR reviewers in your CI pipeline

The writer/reviewer pattern:

Agent writes code → Separate agent reviews → Human final review
Natural second step. Agent catches mechanical issues, human focuses on judgment calls. The writer/reviewer pattern is powerful: two separate agent sessions -- one writes code, another reviews it. The reviewer has fresh context and isn't biased toward code it wrote. For tests: be prescriptive. Encourage minimalist e2e tests over implementation-specific unit tests. ~1min

Use Case 3: Coding

Where most daily work happens

This is not vibe coding -- vibe coding works for prototypes, but production needs structure.

  1. ExploreAgent reads files, understands patterns. No changes.
  2. PlanAgent creates detailed plan. You review and edit it.
  3. ImplementAgent codes against the plan, runs tests.
  4. CommitAgent creates commit, pushes, opens PR.
Stripe ships 1,300+ zero-human-code PRs per week. OpenAI: team of 3-7 built 1M lines, 0 manually-written code, in 1/10th the time. These aren't toy projects -- but they follow this loop.
This four-phase pattern works across all agents. This is not vibe coding. Vibe coding is fine for prototypes, but for production you need structure. The plan is your checkpoint -- review it, edit it, then let the agent execute. The OpenAI stat: a team of 3 growing to 7 built an internal product with ~1M lines, averaging 3.5 PRs per engineer per day -- all agent-written. Use subagents to keep exploration separate from your main context. ~1min

15 Minutes of Planning Saves 2 Hours of Corrections

The secret to high success rates

  • What you want at the end -- requirements, not implementation details
  • Which patterns in the codebase to follow
  • Constraints, edge cases, and files that will change
  • How to verify it works

Vague prompt:

"Add user authentication"

Effective prompt:

"Add JWT auth following the pattern in src/middleware/auth.ts. Include refresh tokens and rate limiting on login. Write integration tests using UserRepository."

This is the single most impactful habit. 15 minutes of planning saves 2 hours of corrections. I often do this in a separate conversation -- the agent and I explore the codebase together, build a plan, and then I paste that plan into a fresh session for implementation. Notice the difference: the vague prompt gives the agent no direction. The specific one points to patterns, states requirements, and tells it how to verify. Don't over-constrain the implementation -- define the what, not the how. ~1min

Give the Agent a Way to Check Its Own Work

  • Unit tests and integration tests
  • Screenshots (Puppeteer, browser automation)
  • A running dev server it can hit
  • Expected output to compare against
  • Multimodal input -- drop in a design mock, give it a dev server, it builds the UI in a loop

Context management -- the core constraint:

  • Clear context between unrelated tasks
  • After 2 failed corrections: start fresh with a better prompt
The single highest-leverage thing: give the agent feedback tools. When it can run tests, take screenshots, check its own work -- success rates jump dramatically. Think of it as guides and sensors: instructions guide the agent BEFORE it acts, tests and linters are sensors that give feedback AFTER it acts. You need both. Context management is critical: clear context between tasks. If you're 2 corrections deep, start a fresh session with a better prompt -- don't pollute the context with failed attempts. ~1min

Use Case 4: PRDs & Engineering Design Docs

Agents as your first draft partner for technical writing

  • Product requirements: Describe what you want in a few sentences → agent generates structured PRD with user stories, acceptance criteria, edge cases
  • The interview pattern: Agent asks YOU questions to extract requirements you hadn't thought of, then drafts the doc
  • Engineering design docs: Agent reads the codebase → drafts architecture, API contracts, trade-offs
  • Deep research: Agent searches the web for prior art and compares approaches before drafting
The first draft is 80% there -- your job is the last 20% of judgment.
Agents are excellent first-draft partners for technical writing. The interview pattern is powerful: instead of you writing the spec, the agent asks you questions -- "What happens if the user loses connectivity mid-checkout? Do you need idempotency?" -- and then drafts the doc from your answers. It surfaces things you hadn't thought about. For engineering design docs, the agent reads the codebase and understands existing patterns before drafting. And it can do deep research -- searching the web for prior art and comparing approaches. The first draft is typically 80% right; your job is the last 20% of judgment and context. ~1min

Use Case 7: Security Review

A security-aware layer on every change

  • PR security scanning: Hardcoded secrets, injection vulnerabilities, insecure API patterns
  • Dependency auditing: Known CVEs, outdated packages with security patches
  • Context-aware analysis: Traces data flow across files -- catches what regex scanners miss
  • IAM review & threat modeling: Overly broad access, new endpoint attack vectors
Unlike SAST tools that pattern-match, agents understand context -- tracing user input through 4 files to a shell command.
Agents as a security layer on every change. The key advantage over traditional security scanning tools is context awareness. A regex-based scanner checks one file at a time. An agent traces user input through your request handler, into a service layer, through a utility function, and into a shell command -- and flags the injection risk. That data flow might span 4 files and wouldn't show up in any single-file scan. Same for dependency auditing: the agent understands whether a CVE actually affects your usage pattern, not just whether the package is present. ~1min

Respect the Exponential

  • The length of tasks AI can handle is doubling every seven months
  • Right now: ~1 hour tasks. Next year: full day. Year after: full week.
  • If we demand reading/writing every line of code, we become the bottleneck
  • The key shift: forget the code, not the product. We stopped reading assembly output. We'll stop reading AI-generated code. But we'll never stop verifying the product works.
  • Managing work you didn't write is not a new problem -- every PM, CTO, and CEO already does this. Engineers are just not used to it yet.
Before we close: this is the exponential we're on. Task complexity AI handles doubles every 7 months. The key insight: forget the code, not the product. We stopped reading assembly when compilers got good enough. We verified at a higher level -- acceptance tests, integration tests, product behavior. The same transition is happening now. And this isn't even a new problem: every PM reviews engineering work without reading every line. Every CTO manages experts in domains they don't fully understand. They verify at the right abstraction level. Engineers are just not used to delegating like this. ~1.5min
